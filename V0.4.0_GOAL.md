# Waycloak v0.4.0 research goal

Last updated: 2026-07-19

## Overall goal

Determine, from primary sources, as-built analysis, target-cluster evidence, and
minimum viable probes, whether eBPF can materially improve Waycloak without
weakening its explicit, fail-closed Kubernetes egress guarantee. Use that
evidence to write the `v0.4.0` PRD and implementation plan.

The value hypotheses are independent: reduced per-Pod privilege, fewer injected
init/sidecar components, improved performance or scaling, and simpler operation.
eBPF is the research focus, not yet an approved product requirement. ADR 0006
and nftables/netlink remain the supported production decision until an
evidence-backed follow-up ADR says otherwise.

The intended compatibility shape is additive: Pod-local nftables/netlink stays
the default for ordinary nodes; a later eBPF mode may be explicitly selected
only for operator-prepared, capability-verified nodes. Unsupported selected
nodes block scheduling or application startup rather than silently falling back.

## Completion criterion

This research goal is complete only when:

- the as-built data-plane responsibilities, component footprint, and privilege
  boundary are mapped;
- viable attachment, lifecycle, ownership, sidecar-reduction, CNI-coexistence,
  and portability models are compared against Waycloak's invariants;
- representative amd64 and arm64 homelab capabilities are measured;
- critical uncertainties are resolved with the smallest safe probes;
- a recommendation, rejected alternatives, support boundary, risks, benchmark
  plan, and decision criteria are published in
  `docs/research/ebpf-data-plane.md`;
- a `v0.4.0` PRD is derived from that evidence, reviewed, and linked to an
  ordered GitHub implementation plan.

Publishing or deploying `v0.4.0` is a later delivery goal defined by the PRD;
it is not a precondition for completing research.

## Research questions

1. Which Waycloak responsibilities can eBPF replace: deny/allow filtering,
   DNS redirection, port translation, policy routing, VXLAN construction,
   health verification, or drift repair?
2. Which attachment model closes traffic before any user init or application
   can run, and remains closed through loader death, restart, upgrade, detach,
   Pod replacement, and node reboot?
3. Can the model reduce per-Pod privilege, injected init/sidecar count, or both?
4. If the component and privilege shape stays similar, does eBPF materially
   improve startup, reconciliation, CPU, memory, throughput, latency, UDP loss,
   or scaling at representative workload counts?
5. Can it coexist with Flannel and other CNIs without owning or flushing their
   qdiscs, programs, maps, links, routes, or cgroups?
6. How are Pod UID, network namespace, veth, and cgroup identities joined
   without races or reuse errors?
7. What kernel, BTF, helper, hook, map, verifier, cgroup, bpffs, security-policy,
   architecture, and runtime capabilities are actually required?
8. Does the measured benefit justify a second security-critical backend or a
   new privileged node component and its support matrix?
9. How do operators prepare, label, verify, schedule, upgrade, and remove eBPF
   capability in heterogeneous clusters without changing the default mode?

## Decision criteria

An implementation PRD may recommend supported eBPF only if the chosen model:

- enforces deny-before-route with no startup or transition escape window;
- covers all IPv4/IPv6, TCP/UDP, DNS, fragmented, and relevant GSO/GRO traffic
  in the documented threat model;
- cannot silently detach or fall back when its loader or controller fails;
- has deterministic, inspectable ownership and safe garbage collection;
- preserves CNI and unrelated host/workload state;
- keeps application containers capability-free and credentials isolated;
- has explicit unsupported-node behavior before application startup;
- passes one backend-neutral black-box packet/lifecycle suite;
- demonstrates material, measured value in privilege, component reduction,
  operability, scale, or performance that outweighs its additional surface.

## Current evidence

- The existing backend installs a Pod-UID-owned nftables output-drop chain
  before validation or routing, then creates the VXLAN link, overlay address,
  policy rules, protected route, DNS redirect, and application-port DNAT.
- Each protected Pod currently receives two Waycloak gating init containers
  and one restartable Waycloak agent sidecar; workload adapters are separate and
  remain necessary only for application-specific lease consumption.
- The prepare, verify, and long-running agent containers use `CAP_NET_ADMIN`;
  application and adapter containers receive no capabilities.
- A filter-only eBPF substitution would retain netlink/VXLAN/routing and
  nftables NAT responsibilities, so it does not inherently remove per-Pod
  `CAP_NET_ADMIN` or injected components. It may still have performance value,
  which must be measured against total system cost.
- A node-owned design could reduce Waycloak's per-Pod component footprint, but
  application-specific adapters such as qBitTorrent's remain a separate concern.
- The homelab has three amd64 Flatcar/Linux 6.12 nodes and two arm64
  Ubuntu/Linux 7.0 Raspberry Pi nodes, all on k3s/containerd with Flannel VXLAN.
- Representative nodes expose cgroup v2, bpffs, vmlinux BTF, privileged
  `bpf()`, BPF JIT, cgroup-BPF, sched-cls, XDP, LWT, hash/array/LPM/ring-buffer
  maps, and tc classifier/action support. This proves mechanisms, not a design.
- The inspected Flannel node had no existing XDP/tc BPF attachments and no
  qdisc filters on `cni0` or `flannel.1`; that is current homelab evidence, not
  a portable CNI assumption.
- A live protected Pod has a UID-addressable systemd cgroup-v2 parent with the
  sandbox and each running container in child `cri-containerd` scopes. This is
  a plausible node-loader identity target; inheritance and startup timing still
  require packet-level proof.
- Disposable amd64 and arm64 Pods passed the first parent-cgroup lifecycle
  probe: each pinned cgroup-skb egress deny remained effective after its loader
  exited, and explicit unpin restored connectivity. QoS-specific path discovery,
  broader traffic coverage, startup timing, replacement, and reboot remain open.
- `CAP_BPF` plus `CAP_NET_ADMIN`, with all other capabilities dropped and
  `RuntimeDefault` seccomp, can load and attach without `privileged: true` on
  both samples. Flatcar also pinned successfully. Ubuntu's enforced default
  AppArmor profile denied pinning; changing only AppArmor to `Unconfined`
  succeeded, proving that node preparation must include a narrow LSM policy.
- One Pod-parent attachment covered ordinary init, restartable init sidecar,
  and application child scopes in a two-phase disposable test.
- Wrong-type link update was rejected without opening traffic; deny-to-deny
  replacement yielded zero successes across 12 bounded connection attempts;
  allow/deny updates and explicit unpin produced the expected state changes.
- Deleting the target cgroup severed its pinned link but left the pin, making
  UID-owned stale-pin garbage collection an explicit lifecycle requirement.
- Both nodes' executable helper inventories show `cgroup_skb` can enforce and
  observe but cannot perform packet rewrite, checksum repair, redirect, or
  tunnel operations. DNS DNAT, lease DNAT, VXLAN, addresses, and routes need a
  separate owner.
- Moving current Pod-netns operations into a node agent requires namespace-entry
  privilege including `CAP_SYS_ADMIN`; CNI-time and host-veth alternatives are
  broader CNI-like architectures, not simple backend replacements.
- A short metrics-server snapshot reported 34-42m CPU and 12-13 MiB for the
  Bitmagnet agent and 43-48m CPU and 12-13 MiB for the qBittorrent agent. The
  current agent runs full configure/verify repair every two seconds, so the
  controlled comparison must include an event-driven or diff-based version of
  the default backend rather than attributing all possible savings to eBPF.
- Required node affinity is ignored after scheduling, so labels can constrain
  initial placement but cannot enforce runtime fail-closed behavior. The design
  needs separate administrator preparation intent, executable observed
  capability, and a node-owned live enforcement condition.
- CNI chained `ADD` is the strongest researched pre-container setup boundary:
  the runtime creates the netns first, runs plugins in order, and halts on
  error. It also introduces runtime-managed binary/config installation and
  explicit DEL/CHECK/GC obligations.
- Scaling a disposable real-provider workload from one to ten sidecar Pods took
  129 seconds and exposed global readiness churn during five Gluetun loopback
  health failures. Deny state remained and all agents recovered. Deterministic
  backend performance must therefore use the fake gateway harness; provider
  membership-burst recovery is a separate system test.
- A reproducible fake-gateway benchmark now measures the complete healthy
  default `Repair`: amd64 Intel N95 median 45.38 ms (36.68-49.64 ms) and arm64
  Raspberry Pi median 7.56 ms (7.20-7.57 ms), with three 20-operation runs on
  each. Both architectures passed the surrounding fail-closed E2E contract.
- containerd 2.2.3 passes CNI the sandbox netns, exact Pod UID labels, and the
  Pod cgroup parent through its `cgroupPath` runtime capability. The target k3s
  node currently chains only Flannel, portmap, and bandwidth, so a Waycloak CNI
  handoff is explicit prepared-node installation and containerd-specific until
  other runtimes are proved.
- The research recommendation is now a developer-preview E2 prototype: chained
  CNI deny-before-start plus a node-owned cgroup eBPF boundary. It is not yet a
  supported backend because dynamic VXLAN/routing/DNS/lease-NAT ownership and
  the broader node-agent threat boundary remain unresolved.
