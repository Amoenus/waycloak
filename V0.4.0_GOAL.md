# Waycloak v0.4.0 release goal

Last updated: 2026-07-19

## Overall goal

Ship `v0.4.0` as the optional eBPF data-plane release: reduce per-workload
networking privilege and overhead where supported, without changing Waycloak's
Kubernetes API or weakening its explicit, fail-closed egress guarantee.

## Completion criterion

The goal is complete only when an explicitly selected eBPF backend passes the
same packet-level fail-closed contract as nftables/netlink on documented amd64
and arm64 environments; final signed `v0.4.0` artifacts are independently
verified, promoted, and observed working in the homelab; and unsupported nodes
fail clearly without silently falling back to another backend.

## Release cutoff

### Backend-neutral contract

- Extract a black-box conformance contract from ADR 0006 covering lockdown,
  configuration, verification, drift repair, detach, upgrade, and cleanup.
- Keep workload/gateway selection, stable identities, allocation handshakes,
  overlay semantics, conditions, events, DNS behavior, and lease APIs backend
  independent.
- Preserve nftables/netlink as the default, supported backend.

### Capability discovery and selection

- Implement a read-only per-node preflight that reports actual kernel config,
  BTF, helper, map, hook, verifier, privilege, lockdown, architecture, and CNI
  compatibility rather than inferring support from kernel version.
- Make eBPF selection explicit in operator-owned configuration and visible in
  admitted workload state and status.
- Reject or prevent scheduling on unsupported nodes before application startup.
- Never fall back silently from selected eBPF to nftables or a permissive path.

### eBPF data plane

- Implement the existing workload-agent data-plane interface with eBPF behind
  an explicit backend selection.
- Install the deny boundary before enabling protected routing and retain it
  through startup, tunnel loss, gateway loss, agent loss, drift, program
  replacement, detach, and upgrade.
- Attach without replacing or flushing CNI-owned programs, maps, routes, hooks,
  or unrelated workload state.
- Use stable Pod/gateway ownership identities for programs, maps, pins, and
  cleanup; partial reconciliation must preserve or strengthen denial.

### Evidence and operational boundary

- Run identical packet-level backend conformance for nftables/netlink and eBPF.
- Test representative amd64 and arm64 nodes and record kernel, BTF, CNI,
  attachment, privilege, and verifier context for every supported result.
- Compare CPU, memory, startup/recovery latency, throughput, UDP loss, and
  steady-state overhead against nftables/netlink at representative workload
  counts.
- Publish scheduling, preflight, troubleshooting, upgrade/rollback, and mixed
  cluster guidance.
- Retain the complete `v0.3.0` real-provider renewal, endpoint rotation,
  ingress, adapter, DHT, gateway-loss, stale-ingress, and same-Pod recovery gate
  for the default backend, plus the applicable protected-egress subset for eBPF.

### Decision gate

- Replace proposed ADR 0019 with an accepted evidence-backed decision only if
  the implementation satisfies conformance, CNI coexistence, privilege, and
  measured-value criteria.
- If the evidence is insufficient, `v0.4.0` must not advertise eBPF support;
  research findings may ship as documentation, but the release goal remains
  incomplete until the stated supported-backend criterion is met or the cutoff
  is explicitly revised.

## Explicitly outside the cutoff

- multiple concurrent gateways, sharding, and cross-gateway failover (`#31`);
- general backup/restore and disaster-recovery expansion (`#32`);
- product-wide metrics, alerts, and dashboards beyond diagnostics required to
  operate and compare the eBPF backend (`#33`);
- broader Kubernetes/CNI certification unrelated to the eBPF support matrix;
- additional VPN engines/providers;
- Loadstone certification and further workload adapters;
- Service-targeted leases, kubectl plugin, UI, Gateway API integration, and
  multi-cluster control planes.

## Maintenance lane

- `#16` and `#64` are repository/CI maintenance and may merge before eBPF work;
  they do not justify a `v0.3.1` runtime release by themselves.
- A `v0.3.1` release is reserved for a user-visible regression or security fix
  against the shipped `v0.3.0` contract.
- `#4` is complete in `v0.3.0`; `#5` must be narrowed to remaining
  Loadstone-only work and removed from the completed release milestone.

## Non-negotiable invariants

- Opt-in remains explicit and visible on the workload Pod template.
- Protected traffic never falls back to ordinary internet egress.
- Selecting eBPF is explicit; unsupported or failed attachment is not a signal
  to use another backend automatically.
- Credentials remain confined to the VPN engine container.
- Application containers gain no Linux capabilities or Kubernetes credentials.
- Stable identities are never derived from alphabetical ordering.
- `Ready` describes observed data-plane health for the selected backend.
- Every externally visible behavior has end-to-end packet-level coverage.

## Progress log

### 2026-07-19

- Audited the shipped architecture, product/API contracts, test strategy,
  roadmap, relevant ADRs, and ten live open GitHub issues.
- Identified issue #65 and proposed ADR 0019 as the governing research and
  decision boundary; the existing `dataplane.Backend` interface provides the
  implementation seam, while ADR 0006 remains the supported default.
- Set the `v0.4.0` cutoff to an explicit, capability-gated eBPF backend with
  nftables parity, no silent fallback, CNI coexistence, amd64/arm64 evidence,
  measured value, signed publication, and homelab proof.
- Deferred unrelated operational-maturity, scale-out, and workload/provider
  expansion so the security-critical backend change has a bounded release gate.
- Reconciled the live tracker: closed completed #4 and the `v0.3.0` milestone;
  narrowed #5 to unmilestoned Loadstone work; removed #16 and #31-#33 from
  completed/current release cutoffs; and made #6, #34, and #65 the three open
  `v0.4.0` milestone issues.
