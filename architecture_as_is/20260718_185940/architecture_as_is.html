Executive summary

  • Waycloak is already a working Kubernetes-native private-egress system, not an early scaffold: Phases 1–4 and releases through v0.2.2 are complete.
  • The product invariant remains selected, fail-closed VPN egress: explicit Pod-template opt-in, no ordinary-egress fallback, no VPN credentials or added application capabilities.
  • The current source and published prerelease are v0.3.0-alpha.6 at the same commit. The latest slice fixed engine auto-healing, unsupported Proton renewal suggestions, and an overly strict public-IP telemetry readiness gate.
  • GitHub comments record live deployment and validation for issues #90, #92, and #94, despite the local roadmap still showing #90 unchecked and the top of PROJECT_STATUS.md still calling live validation a remaining gate.
  • The real remaining v0.3.0 exit work is compatibility proof: sustained qBitTorrent real-provider rotation/renewal with TCP, UDP, tracker, DHT, loss, stale-ingress, and recovery evidence; then completion of Bitmagnet and Loadstone adoption.
  • Operational maturity items (#6, #31–#34, #64, #65, #16) are valid backlog but are ordered after the Phase 5 exit. Multi-gateway sharding and eBPF must not jump the queue.

Entrypoints and runtime

  • Processes: controller/webhook, per-protected-Pod agent, singleton gateway manager beside Gluetun, qBitTorrent adapter, and Bitmagnet adapter.
  • Primary APIs: VPNGateway, controller-owned VPNWorkload, PortForwardLease, cluster-scoped WorkloadAdapter, and the canonical gateway Pod-template annotation.
  • Packaging: Helm is primary; KCL is optional. Runtime does not require Argo CD, Crossplane, ESO, cert-manager, KCL, or Prometheus Operator.
  • Current commands checked: go test ./.... Cluster and release commands were intentionally not run.

Data and integrations

  • Stable overlay allocations live in controller-owned Kubernetes resources; alphabetical renumbering is forbidden.
  • Gateway desired membership and port intents are projected by ConfigMap; the tokenless manager publishes exact applied observations.
  • VPN credentials and native Secret files mount only into the engine. Adapters and protected workloads receive neither VPN credentials nor Kubernetes tokens.
  • External systems are Kubernetes, Gluetun, the configured VPN provider, OCI registries, and GitHub release/signing infrastructure.

Representative flows

  1. Protected startup: annotated Pod → admission generation check/injection → UID allocation ConfigMap → deny-first init → VXLAN/gateway verification → application start. High confidence.
  2. Provider-assigned lease: lease intent → manager NAT-PMP mapping → atomic gateway DNAT → neutral Pod record → adapter applies listener → exact-generation acknowledgement → Ready. High confidence.
  3. Engine failure: Gluetun health fails → readiness withdrawn immediately and traffic remains blocked → delayed liveness restarts only engine container → tunnel/DNS/leases/adapters recover with stable Pod and lease identities. High confidence from implementation history and live #90 evidence.
