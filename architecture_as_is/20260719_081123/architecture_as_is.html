Executive summary

Kubernetes-native control plane
CRDs, controller-runtime reconcilers, and an admission webhook express explicit workload opt-in and observed status.
Deny-first data plane
An injected agent establishes nftables/netlink state before application startup and retains the drop path through gateway loss.
Singleton VPN gateway
A controller-owned StatefulSet runs Gluetun plus a tokenless gateway manager; credentials mount only into the engine.
Renewable ingress
UID-bound leases drive Proton NAT-PMP, stable internal ports, atomic gateway translation, and expiry-aware Pod-local delivery.
Application adapters
Unprivileged qBittorrent and Bitmagnet sidecars consume a neutral record and acknowledge only exact applied generations.
Current release edge
RC8 must prove the NAT-PMP public address and port as one endpoint while reusing the existing provider session.

Entrypoints and runtime

  • Processes: cmd/controller, cmd/agent, cmd/gateway-manager, cmd/qbittorrent-adapter, and cmd/bitmagnet-adapter.
  • Primary API: networking.waycloak.io/v1alpha1 CRDs for VPNGateway, VPNWorkload, PortForwardLease, and WorkloadAdapter.
  • Local runtime contracts: allocation ConfigMap, gateway desired-state ConfigMap, loopback health/observation endpoints, and the versioned adapter lease endpoint.
  • External integrations: Kubernetes API (controller only), Gluetun/OpenVPN, Proton NAT-PMP, OCI registries, and optional cert-manager integration.

Data and integrations

Durable state is Kubernetes-native. Stable overlay allocation, object-UID lease identity, provider internal port, public endpoint generation, and conditions live in CR status and controller-owned ConfigMaps. There is no application database. VPN credentials remain referenced by VPNGateway engine configuration and are not copied into protected workloads or the gateway manager.

The v0.3 endpoint contract is address plus port. Proton supplies both through NAT-PMP; the controller persists both; the agent delivers both; qBittorrent applies the port, overlay binding, and tracker announce address before acknowledgement.

Representative flows

1. Explicit protected egress

Annotated Pod → webhook validates VPNGateway and injects init/agent → controller binds VPNWorkload allocation to Pod UID → agent installs drop state before overlay route → traffic uses gateway VXLAN → gateway forwards only authorized overlay sources through the VPN. Gateway/tunnel loss keeps direct egress blocked. High confidence.

2. Renewable provider-assigned ingress

PortForwardLease UID → stable internal port → gateway manager requests NAT-PMP mapping → public address/port observation → controller generation/status → atomic gateway DNAT → Pod-local neutral record → adapter applies application state → agent/controller observe exact acknowledgement. Expiry or mismatch regresses readiness. High confidence.

3. qBittorrent compatibility

Adapter discovers the Waycloak interface/address, applies qBittorrent listener port and binding, sets the provider public address as announce_ip, restarts enabled DHT when endpoint-visible state changes, verifies the listener, then acknowledges. The real tracker fixture hashes the announced address instead of logging it. High confidence statically; real RC8 proof pending.

Architecture observations

  • Observed: readiness is composite and observation-based; registration alone is insufficient.
  • Observed: provider and application compatibility are deliberately outside generic packet translation.
  • Observed: credential ownership, token isolation, and Linux capabilities are narrow and explicit.
  • Inferred: the strongest remaining v0.3 risk is operational sequencing, not missing core packet enforcement.
  • Unknown: final RC8 behavior until exact digests are promoted and the existing-gateway acceptance completes.
