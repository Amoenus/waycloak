Scope and assumptions

This run examined only the path that propagates an observed singleton gateway Pod endpoint into an existing protected Pod and the path by which the running Linux agent repairs its VXLAN peer. Controller, agent, gateway-manager, port-forward observation, targeted tests, and bounded live-cluster evidence were inspected. Provider credentials and Secret contents were not read. The broader product architecture was not re-audited.

Executive summary

Observed root cause: the allocation ConfigMap was create-once for routing fields, and the agent accepted an existing VXLAN link without comparing its remote endpoint. A replacement gateway therefore left both persisted and kernel state pointing at the former Pod IP.
  • The gateway controller correctly observes the serving gateway Pod IP in VPNGateway.status.overlay.endpoint.
  • The Pod controller previously updated only the port-forward delivery field after allocation creation and did not watch gateway status changes.
  • The agent already polls its projected allocation every two seconds and calls Repair.
  • The Linux repair path previously compared VXLAN VNI and UDP port but not the remote group address, so polling could not repair the stale peer.
  • The gateway manager already polls its desired-state file. A measured rules-generation delay converged without restart and was consistent with kubelet projected ConfigMap latency, not a missing manager reload feature.
  • Fail-closed policy worked: the protected application became unavailable but did not fall back to ordinary egress.

Architecture observations

Observed: desired-state propagation and kernel reconciliation each lacked one half of endpoint replacement. Both are required; fixing only the ConfigMap leaves the immutable VXLAN peer stale, and fixing only the agent leaves it without new desired state.

Inferred: normal gateway manager projection latency is acceptable while provider renewal retains safety margin, but it should remain measured in rotation tests.

Unknown: recovery latency across every supported kubelet ConfigMap sync configuration and CNI remains unverified.
