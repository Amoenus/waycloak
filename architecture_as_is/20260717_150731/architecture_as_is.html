  • Scope: Waycloak control/data-plane behavior for the live Proton/OpenVPN gateway and qBittorrent consumer, plus its homelab GitOps source.
  • Execution: repository inspection, live read-only Kubernetes diagnostics, upstream issue/release research, GitOps engine-image remediation, and controlled singleton replacement.
  • Not explored: credential values, provider account state, unrelated homelab workloads, and additional VPN engines.

Executive Summary

Outcome: Waycloak behaved fail closed. The original non-recovery was an upstream Gluetun v3.41.0 health-check hang. Gluetun v3.41.1 was deployed, after which the replacement reached the provider but was independently rejected with OpenVPN AUTH_FAILED.
  • The gateway manager immediately withdrew tunnel, overlay, DNS, port-forward, and overall readiness when Gluetun stopped.
  • The workload agent and adapter withdrew readiness while the qBittorrent process remained running; no application container restart or direct-egress fallback was observed.
  • Desired and applied membership generations initially matched, excluding stale membership as the original trigger.
  • Gluetun v3.41.0 logged a health-check-driven VPN restart and stopping, but no later VPN start for many hours.
  • Upstream v3.41.1 explicitly fixes a health-check race that could hang Gluetun completely.
  • The homelab GitOps source now pins v3.41.1 by immutable OCI index digest.
  • The replacement gateway changed underlay endpoint as expected; the existing qBittorrent Pod retained its UID and zero restarts.
  • Final tunnel recovery is externally blocked by rejected provider credentials, owned by a separate rotation task.

Entrypoints And Runtime

  • Control-plane entrypoints: cmd/controller, admission webhook, Kubernetes reconciliation watches.
  • Gateway runtime: controller-owned singleton StatefulSet using OnDelete.
  • Workload runtime: injected init gates plus a long-running routing agent; application containers receive no added capabilities.
  • GitOps entrypoint: kcl/frontend/networking/routed_vpn_gateway.k in the homelab repository.

Data And Integrations

  • Kubernetes CR state owns stable gateway, workload, membership, and lease identities.
  • Operator-owned ConfigMap supplies non-secret Gluetun-native settings.
  • Engine-only Secret file projection supplies OpenVPN credentials; values were not inspected.
  • External systems: Proton/OpenVPN, Docker Registry, GitHub release and issue metadata, ArgoCD.

Test Suite Interpretation

  • Waycloak unit, envtest, Kind, and real-provider harnesses establish fail-closed, endpoint-replacement, and lease-delivery contracts.
  • Focused homelab KCL rendering proved the new immutable Gluetun identity.
  • Server-side networking validation and Markdown lint passed.
  • The full serial KCL source gate was stopped after reaching the qBittorrent source because it linted every unrelated application; targeted affected-source checks had passed.

Representative Flows

  1. Original failure: periodic engine health failure → Gluetun stops tunnel → manager observation fails → all gateway readiness components regress → provider lease expires → workload agent and adapter regress → traffic remains blocked. High confidence.
  2. GitOps remediation: v3.41.1 digest committed and merged → ArgoCD applies VPNGateway → controller reconciles OnDelete template → operator deletes exact singleton Pod → StatefulSet recreates it from the new image. High confidence.
  3. Replacement startup: v3.41.1 starts → reaches provider endpoint → provider returns AUTH_FAILED → no tunnel interface → manager remains unready → qBittorrent remains fail closed with stable UID. High confidence.

Architecture Observations

  • Observed: the readiness cascade reflects real data-plane health rather than registration.
  • Observed: the engine image is operator-owned native configuration, so upgrading Gluetun does not require a Waycloak release.
  • Inferred: the v3.41.0 stop-without-restart is the upstream health-check race because the live sequence matches and v3.41.1 names that fix.
  • Unknown: why the provider now rejects the existing credentials; account/credential inspection is outside this task.
