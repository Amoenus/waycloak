Executive Summary

Control plane
One controller-runtime manager hosts four reconcilers and admission webhooks with leader election and an observed admission-generation gate.
Data plane
Per-workload native nftables/netlink agents establish deny-first VXLAN paths to a tokenless singleton gateway manager.
Provider boundary
Gluetun owns VPN establishment and native provider configuration; Proton NAT-PMP is a capability driver behind a provider interface.
Lease boundary
Manager-owned mapping generations separate endpoint changes from expiry-only renewals; Kubernetes status remains controller-owned.
Extension boundary
Trusted, digest-pinned, unprivileged workload adapters consume a versioned Pod-local protocol outside the core controller.
Planning implication
The existing backend interface and conformance-first threat model support a focused eBPF release, provided selection is explicit and unsupported nodes never fall back silently.

Entrypoints And Runtime

  • cmd/controller/main.go: controller manager, webhook, health/readiness, leader election.
  • cmd/agent/main.go: injected prepare/verify/repair lifecycle inside each protected Pod network namespace.
  • cmd/gateway-manager/main.go: tokenless manager beside the VPN engine.
  • cmd/qbittorrent-adapter and cmd/bitmagnet-adapter: application-specific protocol endpoints outside core reconciliation.
  • Helm is the primary install surface; plain CRDs and Pod annotations are canonical; KCL is optional.

Data And Integrations

  • Kubernetes API objects own durable intent and observation: VPNGateway, controller-owned VPNWorkload, PortForwardLease, and cluster-scoped WorkloadAdapter.
  • ConfigMap projection carries stable desired membership/lease intent and restart hints, not fast provider-event signaling.
  • The gateway manager and adapters have no Kubernetes token. VPN credentials mount only into the engine.
  • Gluetun and the VPN provider are external systems; provider-specific capability stays behind interfaces.

Test Suite Interpretation

  • Unit and envtest suites cover mutation, stable allocation, conditions, provider state machines, engine-native validation, and ownership.
  • Kind/k3s tests supply packet-level fail-closed, DNS, VXLAN, nftables, upgrade, and adapter evidence.
  • The gated real-provider suite uniquely proves Proton renewal/rotation, public TCP/UDP ingress, qBittorrent tracker/DHT behavior, gateway loss, and same-Pod recovery.
  • No performance baseline or multi-version Kubernetes/CNI matrix is yet published; those remain issue #34.

Representative Flows

  1. Protected startup. Annotation enters admission; generation and authorization are checked; init installs lockdown before route setup; allocation and gateway health must converge before application startup. High confidence.
  2. Provider endpoint rotation. Manager acquires a changed mapping, advances its local generation, installs matching rules, exposes observation; controller publishes status/delivery; adapter applies and acknowledges; readiness returns only after all observations match. High confidence.
  3. Gateway loss. Agent lockdown remains, gateway/lease readiness regresses, ordinary internet egress stays blocked, replacement endpoint is propagated and repaired without changing the workload identity. High confidence.
  4. Upgrade. Helm generation changes gate webhook readiness; control plane rolls before protected workloads; gateway template changes remain OnDelete and operator-activated. Backup/restore and full disaster-recovery semantics remain incomplete. Medium confidence for the unimplemented recovery extensions.

Architecture Observations

Observed

  • Waycloak already follows the Kubernetes operator pattern through level-based controller-runtime reconcilers, observed conditions, and owner references.
  • The supported topology is one stateful tunnel Pod per declared gateway; current release evidence uses one gateway and one provider session.
  • The backend interface exists, but nftables/netlink is the only accepted production implementation.

Inferred

  • An eBPF implementation can preserve the public API by implementing the existing backend seam, but capability discovery, attachment ownership, and packet-level parity are release-critical.
  • Multi-gateway sharding, general lifecycle/DR, and product-wide observability are independent expansions and should not share the eBPF cutoff.

Unknown

  • Actual capacity limits at 10/50 protected workloads and across supported CNI/kernel combinations.
  • Restore semantics for persisted stable identities after partial or total control-plane loss.
  • Viable eBPF hooks and feature availability on target amd64/arm64 nodes.
