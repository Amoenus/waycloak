# ADR 0007: Transparent gateway-routed DNS

Status: Accepted
Date: 2026-07-13

## Context

Protected Pods must retain Kubernetes search domains and service discovery, but
external DNS must not reach a resolver outside the selected gateway. Admission
does not yet know the Pod's allocated overlay address, so it cannot write the
gateway resolver into `dnsConfig`. A localhost proxy in the conventional agent
would also race application startup, and replacing the Pod resolver policy
would discard kubelet-generated search configuration.

## Decision

The workload agent transparently destination-NATs all IPv4 and IPv6 UDP and TCP
port 53 traffic to the gateway overlay address on port 53. The rules live in the
same Pod-UID-owned nftables table as the fail-closed filter and are reconciled
atomically with it. Applications keep the resolver configuration generated by
kubelet, including search domains, but the packet destination cannot bypass the
gateway resolver.

The gateway resolver forwards configured Kubernetes cluster zones to cluster
DNS and resolves external names through an upstream path inside the protected
gateway. Direct DNS fails when the gateway or protected path is unavailable.
Encrypted DNS is not identified by protocol; it follows the same protected
default route as all other external TLS or QUIC traffic.

## Consequences

- UDP DNS, TCP fallback, search-domain resolution, and gateway outage require
  packet-level acceptance tests.
- The gateway needs a DNS listener on its overlay address and narrowly scoped
  bind permission for port 53.
- `Preserve` mode can retain direct cluster traffic without allowing direct
  cluster-DNS bypass.
- Non-standard DNS ports are ordinary protected traffic; they are not treated
  as trusted resolver exceptions.

## Alternatives rejected

- Replacing `dnsConfig` during admission: the allocation and gateway overlay
  address do not exist yet, and cluster search domains are installation-specific.
- A localhost agent proxy: conventional sidecar startup is not ordered before
  application startup.
- Allowing cluster DNS directly in `Preserve` mode: external upstream behavior
  would be outside the selected gateway and could leak during VPN failure.
- Blocking port 53 without a gateway resolver: breaks Kubernetes discovery and
  does not satisfy the product experience.
