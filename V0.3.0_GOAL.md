# Waycloak v0.3.0 release goal

Last updated: 2026-07-19

## Completion criterion

The goal is complete only when final `v0.3.0` immutable artifacts are published,
independently verified, promoted to the homelab GitOps repository, and observed
running successfully in the homelab cluster.

## Release cutoff

`v0.3.0` must implement and prove a level-based, observable renewable
port-forward reconciliation model that:

- separates provider endpoint changes from expiry-only lease renewals;
- preserves explicit, fail-closed Kubernetes egress during every transition;
- never reports a port-forward lease Ready unless the provider mapping,
  gateway rules, exact workload target, delivery, and any required application
  acknowledgement are current and observed;
- tolerates normal asynchronous renewal convergence while an already-observed
  mapping remains valid;
- withdraws readiness before a mapping becomes stale, on endpoint rotation
  until the new path is observed, and during tunnel, gateway, agent, or adapter
  loss;
- provides enough durable status and test diagnostics to identify which
  reconciliation stage is pending or failed.

## Authorization and operating assumptions

- Direct commits and pushes to `main` are authorized.
- Waycloak, its gateway, and opted-in applications may be deleted, restarted,
  disrupted, or otherwise manipulated as required in the dev/test homelab.
- Argo CD hard refreshes and syncs are authorized.
- Only the explicitly selected Waycloak gateway and dependent workloads are in
  disruption scope; unrelated workloads and user-owned untracked architecture
  reports remain untouched.
- The single-provider-session constraint remains: tests reuse the existing
  Proton/OpenVPN gateway and must not create a competing credential session.

## Non-negotiable invariants

- Opt-in remains explicit on the workload Pod template.
- Protected traffic never falls back to ordinary internet egress.
- Provider credentials remain confined to the VPN engine container.
- Application containers receive no additional Linux capabilities.
- Stable workload identities, overlay allocations, internal port identities,
  and lease identities are not alphabetically renumbered.
- `Ready` represents observed data-plane state.
- Every externally visible behavior added or changed has acceptance coverage.

## Current evidence

- `v0.3.0-rc.9` is published, independently verified, and deployed in the
  homelab.
- RC9 fixed the duplicate-overlay-allocation race by allocating against an
  authoritative API-server read; simultaneous real workload recreation then
  produced distinct overlay addresses.
- The real-provider acceptance test proved initial protected egress,
  provider-backed ingress, tracker configuration, DHT participation, gateway
  rule installation, and lease delivery.
- A later renewal failed to regain complete lease readiness within two minutes.
- Observed evidence included an expiry mismatch between the refreshed provider
  observation and Pod delivery acknowledgement, plus transient gateway health
  observations. The isolated test resources were cleaned up before the exact
  terminal condition was preserved.
- The existing controller uses controller-runtime/Kubebuilder patterns,
  Kubernetes watches for API resources, and a two-second `RequeueAfter` poll
  for external gateway and Pod-local observations.

## Proven failure and design direction

The instrumented reproduction identified the stalled stage. The provider
mapping expired after a short lifetime, observation briefly became unavailable,
and a newly acquired endpoint advanced the lease generation. Provider
observation recovered, but `GatewayRulesReady` remained false across the full
two-minute wait and another renewal.

The controller currently derives `LeaseGeneration` from the observed public
endpoint, writes that generation into the gateway desired-state ConfigMap, and
waits for the gateway manager to install matching rules. The gateway manager
reads that ConfigMap through a mounted volume. Kubernetes documents that a
mounted ConfigMap update may take a kubelet sync period plus cache propagation
delay to reach a container. A short-lived provider lease can therefore rotate
again before the previous generation reaches the manager, leaving gateway
rules perpetually one generation behind.

The design direction is to remove this circular observation-to-desired-state
feedback path. Stable Kubernetes intent remains in the ConfigMap, but the
gateway manager must acquire/renew the provider mapping and install the matching
rule generation locally in one reconciliation boundary. The controller then
observes and publishes that manager-owned mapping generation. Expiry-only
renewal remains distinct from endpoint rotation, and readiness remains
fail-closed whenever the currently valid mapping lacks matching observed rules.

## Progress log

### 2026-07-19

- Reframed final release around level-based operator reconciliation rather than
  treating a longer test timeout as sufficient.
- Confirmed Waycloak already uses controller-runtime and Kubebuilder operator
  patterns; adopting another SDK is not the expected fix.
- Identified expiry equality within an unchanged lease generation as the first
  concrete boundary to instrument and test.
- Added a bounded real-provider lease transition timeline containing mapping
  generation, renewal timestamps, and condition transitions while deliberately
  omitting provider endpoint values. Added and passed a regression proving the
  diagnostic cannot expose the provider address or port.
- Reproduced the failure with the new timeline. A short-lived mapping expired,
  a new mapping advanced `LeaseGeneration`, and provider observation recovered,
  while `GatewayRulesReady` remained false throughout the two-minute bound.
- Correlated the failure with the current circular flow: manager observation ->
  controller status generation -> desired ConfigMap -> kubelet volume projection
  -> manager rule generation. Confirmed Kubernetes-mounted ConfigMap propagation
  is not a suitable signaling path for leases with comparable lifetimes.
- Accepted ADR 0023. The gateway manager now owns monotonic mapping generation
  and installs matching rules from the same local observation. ConfigMap status
  generation is only a restart seed, and controller observations cannot regress
  it.
- Separated expiry-only renewal from mapping rotation. An unexpired mapping
  remains ready during retry and exposes `renewalPending`; expiry or tunnel loss
  zeroes the effective rule generation. Same-generation delivery may carry an
  earlier unexpired deadline, but never one later than provider expiry.
- Added unit/state-machine regressions for renewal preservation, endpoint
  rotation, local rule convergence without ConfigMap refresh, generation
  regression rejection, delivery deadline safety, and fail-closed expiry.
- Added the previous public endpoint to the restart hint and reject endpoint
  changes that reuse an existing generation. Once a stale projected hint
  catches up, the manager advances the current endpoint without another
  provider request, preserving both monotonicity and automatic recovery.
- Advanced chart and KCL source metadata to `v0.3.0-rc.10` for certification.
- Pushed commit `6df95d3` to `main`. The first CI run passed Linux race,
  static, generated-artifact, Helm, and pinned KCL gates, then exposed two
  stale test fixtures: envtest omitted the now-authoritative manager generation,
  and the Kind DNAT fixture incorrectly pre-seeded generation 1 before first
  acquisition. Updated both fixtures to model manager-owned generation 1 and
  endpoint rotation generation 2.
- Reran the corrected six-minute DNAT/VXLAN acceptance directly against the
  homelab Kubernetes path. It passed initial TCP/UDP forwarding, manager-owned
  rotation, source-port translation, second-identity add/remove stability,
  malformed-projection retention, and membership recovery without using a VPN
  credential or creating a competing provider session.
